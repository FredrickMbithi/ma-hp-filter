# Backtest Engine Specification

**Status:** Implemented  
**File:** `src/backtest/engine.py`  
**Date:** March 2026

---

## 1. Core Design Principle

The backtest engine simulates how trades actually happen:

```
Market Data ŌåÆ Signal ŌåÆ Position ŌåÆ Trades ŌåÆ Costs ŌåÆ Equity
```

**The engine is strategy-agnostic.**  
It accepts any signal Series and any cost model. It has zero knowledge of
moving averages, HP filters, z-scores, or any indicator. If the engine
references an indicator, the design is broken.

Correct dependency graph:

```
data ŌåÆ strategy ŌåÆ signal ŌåÆ engine ŌåÆ results
```

Wrong:

```
data ŌåÆ engine (with embedded MA/HP logic) ŌåÆ results
```

---

## 2. Execution Model

### Signal execution timing

| Time        | Event                                       |
| ----------- | ------------------------------------------- |
| Bar T close | Signal generated by strategy                |
| Bar T+1     | Position takes effect; return earned on T+1 |

This is enforced in code via:

```python
positions = signal.shift(1).fillna(0)
```

**No exceptions.** Signals generated at bar T cannot fill at bar T.
This would constitute lookahead bias ŌĆō the most common source of fake alpha
in backtesting.

### Implementation note

This specification uses close-to-close returns as the return series. The
execution lag (TŌåÆT+1) is modelled as a 1-bar shift of the signal. A more
precise model would use T+1 open price for execution and T+1 close for the
bar's return attribution, but the close-to-close approximation is adequate
for H1 data where open Ōēł previous close.

## 3. Data Requirements

| Property                 | Requirement                                                   |
| ------------------------ | ------------------------------------------------------------- |
| Index type               | `pd.DatetimeIndex`, UTC-localized                             |
| Required columns         | `open`, `high`, `low`, `close`, `volume`                      |
| Frequency                | Any consistent timeframe (H1 primary)                         |
| No duplicates            | Enforced by `FXDataLoader.validate()`                         |
| No gaps                  | Fill with `FXDataLoader.fill_gaps()` before passing to engine |
| Monotonically increasing | Required; enforced by loader                                  |

Minimum columns required by the engine: **`close` only**.
The engine only uses `data["close"]` for return computation.
OHLC columns are passed through as context but not consumed.

---

## 4. Signal Convention

Signals represent desired exposure direction. They are not positions.

| Value | Meaning            |
| ----- | ------------------ |
| +1    | Long signal        |
| 0     | Flat (no position) |
| ŌłÆ1    | Short signal       |

Fractional signals (e.g. `0.5`) are allowed for volatility-scaled sizing.

Signals are produced by the strategy layer and must have the same
`DatetimeIndex` as `data`.

---

## 5. Position Model

The position is the execution-lagged realization of the signal:

```python
positions = signal.shift(1).fillna(0)
```

| Bar | Signal | Position |
| --- | ------ | -------- | ------------------------------------------ |
| T=0 | 1      | 0        | ŌåÉ no position yet (shift fills NaN with 0) |
| T=1 | 1      | 1        | ŌåÉ long entered at T=1                      |
| T=2 | 1      | 1        | ŌåÉ still long                               |
| T=3 | ŌłÆ1     | 1        | ŌåÉ short signal, long position still active |
| T=4 | ŌłÆ1     | ŌłÆ1       | ŌåÉ position is now short                    |
| T=5 | 0      | ŌłÆ1       | ŌåÉ exit signal, still short until T+1       |
| T=6 | 0      | 0        | ŌåÉ flat                                     |

---

## 6. Trade Definition

A trade is a **change in position**:

$$\text{trade}_t = \text{position}_t - \text{position}_{t-1}$$

```python
trades = positions.diff().fillna(0)
```

| Bar | Position | Trade | Interpretation                   |
| --- | -------- | ----- | -------------------------------- |
| T=0 | 0        | 0     | No activity                      |
| T=1 | 1        | +1    | Entered long                     |
| T=2 | 1        | 0     | Held long                        |
| T=3 | 1        | 0     | Held long                        |
| T=4 | ŌłÆ1       | ŌłÆ2    | LongŌåÆshort flip (2 units traded) |
| T=5 | ŌłÆ1       | 0     | Held short                       |
| T=6 | 0        | +1    | Closed short                     |

**Critical note:** A long-to-short flip costs **twice** a normal entry or exit.
`|trade| = 2` means two units change hands. Cost models see `abs(trade) = 2`
and charge accordingly. Many implementations miss this.

---

## 7. Returns

### Market return

```python
returns = data["close"].pct_change().fillna(0)
```

### Gross strategy return

```python
strategy_returns = positions * returns
```

### Net return

```python
net_returns = strategy_returns - costs
```

### Equity curve

```python
equity = (1 + net_returns).cumprod()
```

The equity curve starts at 1.0. Multiply by initial capital for dollar P&L.

---

## 8. Cost Model Interface

Cost models must implement a single method:

```python
def compute(self, trades: pd.Series) -> pd.Series:
    """
    Parameters
    ----------
    trades : pd.Series
        Position diff series (signed; e.g. -2 for longŌåÆshort flip).

    Returns
    -------
    pd.Series
        Non-negative cost per bar as a decimal fraction.
        Same index as trades.
    """
```

### Available implementations

| Class         | Location                     | Description                                          |
| ------------- | ---------------------------- | ---------------------------------------------------- |
| `ZeroCost`    | `src/backtest/engine.py`     | No-op; returns zero. Used for gross-return analysis. |
| `FXCostModel` | `src/backtest/cost_model.py` | Fixed spread + slippage in bps.                      |

### Extension pattern

New cost models do **not** inherit from a base class. Python's structural
subtyping (`Protocol`) is used. Any object with `compute(trades) -> Series`
satisfies the interface:

```python
class MarketImpactCost:
    def __init__(self, impact_bps_per_lot: float):
        self.impact = impact_bps_per_lot

    def compute(self, trades: pd.Series) -> pd.Series:
        return trades.abs() * self.impact / 10_000
```

## 9. Engine Output

`VectorizedBacktest.run()` returns a `dict[str, pd.Series]` with the
following keys:

| Key                | Type        | Description                                   |
| ------------------ | ----------- | --------------------------------------------- |
| `equity`           | `pd.Series` | `(1 + net_returns).cumprod()` starting at 1.0 |
| `strategy_returns` | `pd.Series` | Gross returns before costs                    |
| `net_returns`      | `pd.Series` | Returns after costs                           |
| `costs`            | `pd.Series` | Per-bar cost (non-negative, decimal fraction) |
| `positions`        | `pd.Series` | Signal shifted by 1 bar                       |
| `trades`           | `pd.Series` | Position diff (entry/exit/flip events)        |

**Metrics are not computed here.** Sharpe ratio, max drawdown, Calmar ratio,
win rate, profit factor, and turnover belong in a future `src/backtest/metrics.py`.

## 10. Engine Usage Example

```python
import yaml
from src.data.loader import FXDataLoader
from src.features.hp_filter import apply_hp_filter, cycle_zscore
from src.features.moving_average import sma
from src.backtest import VectorizedBacktest, FXCostModel

# 1. Load data
loader = FXDataLoader("data/raw")
df = loader.load("USDJPY_10yr_1h")

# 2. Generate signal (strategy layer ŌĆō outside the engine)
_, cycle = apply_hp_filter(df["close"])
zscore = cycle_zscore(cycle)
ma_fast = sma(df["close"], 50)
ma_slow = sma(df["close"], 200)

signal = (
    (zscore < -2.0).astype(float) * (ma_fast > ma_slow).astype(float)  # long
    - (zscore > 2.0).astype(float) * (ma_fast < ma_slow).astype(float)  # short
)

# 3. Build cost model
with open("config/config.yaml") as f:
    cfg = yaml.safe_load(f)
cost = FXCostModel(
    spread_bps=cfg["execution"]["commission_bps"],
    slippage_bps=cfg["execution"]["slippage_bps"],
)

# 4. Run backtest
bt = VectorizedBacktest(data=df, signal=signal, cost_model=cost)
results = bt.run()

# 5. Inspect results
print(results["equity"].iloc[-1])  # terminal equity
print(results["trades"].abs().sum())  # total units traded
```

## 11. What This Engine Does NOT Do

The following are explicitly out of scope for `VectorizedBacktest`:

- **Walk-forward / out-of-sample splits** ŌĆō use `safe_train_test_split()`
  from `src.data.loader` before passing data.
- **Parameter optimization** ŌĆō run the engine N times externally.
- **Portfolio construction** ŌĆō multi-asset signals, volatility scaling, risk
  parity belong in a future `src/portfolio/` layer.
- **Stop-loss / take-profit orders** ŌĆō not modellable in vectorized mode.
  These require the event-driven engine (future work).
- **Partial fills / order books** ŌĆō event-driven engine only.
- **Performance metrics** ŌĆō future `src/backtest/metrics.py`.

## 12. Vectorized vs Event-Driven

| Dimension            | Vectorized (current) | Event-Driven (future) |
| -------------------- | -------------------- | --------------------- |
| Speed                | Fast (seconds)       | Slow (minutes)        |
| Realism              | Moderate             | High                  |
| Order types          | Market only          | Limit, stop, bracket  |
| Implementation       | pandas/numpy         | Bar-by-bar loop       |
| Lookahead protection | `signal.shift(1)`    | Inherent by design    |
| Use case             | Research, iteration  | Final validation      |

**Use the vectorized engine for all research-stage work.**
Only migrate to event-driven for final pre-live validation.
